<task type="auto" tdd="true">
  <name>Task 1: Add Phase 2 board schema and migrations</name>
  <files>server/schema.sql, server/store.py</files>
  <behavior>
    - Fresh databases include `events.parent_event_id`.
    - Existing live databases are migrated idempotently by `init_db`.
    - Acknowledgements are stored in a dedicated append-only table.
    - Ack rows include `ack_id`, `message_id`, `agent_id`, and `delivered_at`.
    - Update/delete triggers protect ack rows the same way event rows are protected.
  </behavior>
  <action>
    Add `parent_event_id TEXT REFERENCES events(event_id)` to the canonical
    `events` table.

    Add a `message_acks` table:
    - `ack_id TEXT PRIMARY KEY`
    - `message_id TEXT NOT NULL REFERENCES events(event_id)`
    - `agent_id TEXT NOT NULL`
    - `delivered_at TEXT NOT NULL`
    - unique constraint on `(message_id, agent_id)` so delivery proof is
      idempotent per agent/message pair

    Add append-only triggers for `message_acks`.

    Add a small `init_db` migration helper that checks existing table columns
    with `PRAGMA table_info(events)` and applies `ALTER TABLE` only when needed.
  </action>
</task>

<task type="auto" tdd="true">
  <name>Task 2: Store operations for ack and reply</name>
  <files>server/store.py, server/schema.py</files>
  <behavior>
    - `ack_message(con, message_id, agent_id)` validates that the target event
      exists, writes one ack record, commits, and returns a receipt.
    - Ack receipt shape is small and deterministic:
      `{ack_id, message_id, agent_id, delivered_at, validation_status}`.
    - Re-acking the same message by the same agent is idempotent: return the
      existing ack receipt with `already_acknowledged: true`.
    - `reply_message(con, thread_id, parent_message_id, agent_id, kind,
      content_md)` validates parent existence and same-thread ownership.
    - Reply receipt uses the existing event receipt shape and returns the new
      event id.
  </behavior>
  <action>
    Add ack validation in `schema.py` using the same Draft7Validator pattern as
    `EVENT_SCHEMA`.

    Extend `EVENT_SCHEMA` with optional `parent_event_id`.

    Implement `ack_message` and `reply_message` in `store.py`; keep helper
    functions narrow and local to the store layer.
  </action>
</task>
